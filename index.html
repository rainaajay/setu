Setu.

A settlement system without blockchain.

What it is

Setu is a working prototype of a payment and trust system that keeps what fifteen years of blockchain research proved durable and drops the rest. There are no blocks, no mining, no fee market and no token. A payment is final the moment a quorum of authorities has countersigned it — one network round trip. Read the white paper →

The design rests on one finding: payments do not need a single global ordering of all transactions. Only spends from the same account can conflict. Removing global ordering removes the queue, the fee auction, and the extractable value that comes with them.

Who needs that? Software agents. A human tolerates card fees and settlement delays; an agent buying ten thousand API responses cannot. Setu is a settlement rail built for machine-to-machine commerce, with humans holding the delegation keys.

Built for agents

  • Delegation. Your key signs one credential: "this agent may spend 10 units, at most 2 per payment, until this expiry." Any merchant verifies it offline. The agent transacts; you hold the leash.
  • Pay-per-request. A merchant agent answers with an invoice; the buyer agent pays it on the network with the invoice id signed into the order; the settlement certificate is the receipt, verifiable by anyone with the committee's public keys. No accounts, no API keys, no card on file.
  • Oversight by sentinels. Audit agents continuously compare all four regions and flag any divergence or misbehaving authority. The back office is software.

Measured on the live network: a buyer agent made 10 delegated, invoice-bound micropayments averaging 186 ms finality each, was stopped by its budget policy on the 11th, a forged certificate was rejected, and the sentinel found every region consistent.

Connect an agent

Any agent can use Setu without Setu-specific code, through the protocols agents already speak:

  • MCP — load the Setu server and your agent gets a wallet and can pay or charge. One line of client config:
    { "mcpServers": { "setu": { "command": "node", "args": ["…/packages/setu-mcp/server.ts"] } } }
  • x402 — a live pay-per-call endpoint any agent can pay over HTTP: setu-gateway.fly.dev/premium-quote returns 402 with payment instructions; pay on Setu, retry, get the resource.
  • Discovery — a machine-readable agent card: /.well-known/agent-card.json.

A stranger agent can discover, pay for, and consume that endpoint with no account and no API key — demonstrated end to end (packages/setu-gateway).

How it is built

  • Plane 1 — settlement. A small committee countersigns transfer orders. Two conflicting spends can never both collect a quorum, so double-spending is impossible rather than merely expensive. Demonstrated against a Byzantine authority that signs everything.
  • Plane 2 — trust. Identities and attestations live in a peer-to-peer hash-linked log with no committee and no server. Peers can work offline and converge on reconnect. Forged and tampered records are rejected by every honest peer; a peer that tells two parties two different stories cannot split the network.
  • One key space. A payment address is also an identity that can be attested — "pay this counterparty" and "who vouches for this counterparty" are views of the same key.

Deliberately given up: general smart contracts (proven incompatible with consensus-free settlement) and a speculative native token.

Research base

  • FastPay — Byzantine settlement without consensus; 80,000 TPS, sub-100 ms finality.
  • pod — one round trip is the physically optimal confirmation latency.
  • ABC — permissionless ledgers need neither proof-of-work nor consensus; the cost is general smart contracts.
  • Kleppmann, PaPoC 2022 — replicated data structures that tolerate unlimited Byzantine peers.
  • IOTA post-mortem — why the feeless DAG failed: economics, not technology. Setu prices spam with rate limits, not fees.
