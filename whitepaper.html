White Paper · v3.1

Setu. A consensusless settlement engine for prefunded agent payments.

Low-latency, policy-controlled settlement for machine-to-machine commerce — anchored to existing money, exposed through existing agent protocols.

v3.1 · 18 July 2026 · Working prototype live across four regions on three continents (London, Frankfurt, Washington, Singapore) · v3.1 adds the wide-area chained-spend measurement (§6) · Implementation note & acknowledgements in §11.

Executive proposition

What Setu is. A settlement engine that moves a prefunded balance between accounts in one network round trip, using a small committee of named authorities that countersign transfers — no global consensus, no blocks, no mining, no native Setu token, and no mandatory per-transaction network fee. Its distinguishing feature is server-enforced delegated budgets: a person authorises an agent to spend under a cumulative limit that the authorities themselves track and enforce.

What runs today. Four authorities live across four regions; a browser wallet; a public explorer; adversarial safety tests; a chained-spend benchmark; and a working delegation-allowance protocol. Details and limits in §6.

The precise, bounded claim. The core settlement protocol is not novel — it is the FastPay family of consensusless-payment research (§3). Setu's contribution is a lightweight working implementation, a server-enforced delegation layer, and a commercialisation aimed at the market that did not exist when that research was written: delegated machine-to-machine commerce. Setu is settlement infrastructure that must anchor to an existing regulated form of money (§7) and interoperate with existing agent-payment standards (§8). It is not, and does not aim to be, a new currency or a universal settlement network.

01The use case: why agent payments differ

A human tolerates a card fee and next-day settlement. A software agent buying ten thousand data responses cannot.

Machine-to-machine commerce inverts the assumptions retail payments are built on: values are often sub-cent, frequency is high and continuous, there is no human in the loop to authorise each payment, and the "merchant" is frequently an API returning a result in milliseconds. Card authorisation economics, chargeback windows, and per-transaction fees that are rounding errors for a £40 purchase become prohibitive at a hundred payments per second of a tenth of a penny each. This is the requirement gap Setu targets — and it is a genuinely new market, not a better version of an old one.

02Why global ordering is unnecessary for simple payments

Many blockchain systems impose a single global order over every transaction. For simple single-owner asset transfers, that is more than the problem requires: a payment from Alice conflicts only with another payment Alice makes with the same funds — never with unrelated payments elsewhere. Ordering is needed only among debits from the same account.

This is a formal result, not a slogan. Guerraoui et al. proved that a single-owner asset-transfer object has consensus number one — it can be implemented without consensus. The costs people associate with public blockchains follow specifically from imposing more structure than that: a single global queue creates a fee market to enter it; a block producer's discretion over ordering creates extractable value (MEV); and proof-of-work security spends energy to make an anonymous global history expensive to rewrite. Remove the global-ordering requirement for payments and those specific costs go with it. This is a narrow, defensible claim — it says nothing about permissioned ledgers, rollups, or DAGs that make different trade-offs.

03Prior work, and Setu's contribution

The consensusless-payment approach is well developed, and Setu should be read as a member of that family, not a departure from it:

  • FastPay (Baudet, Danezis, Sonnino; Meta/Novi, 2020) is the direct antecedent of Setu's settlement plane: pre-funded accounts, a committee of authorities, quorum-signed transfer certificates, Byzantine consistent broadcast, no consensus, sub-100 ms confirmation.
  • Astro (Collins et al., DSN 2020) implements payments by reliable broadcast; Zef (2022) extends FastPay to private payments via anonymous credentials and zero-knowledge proofs — the direction Setu's privacy roadmap points.
  • The theory sits under all of them: single-owner transfer has consensus number one (Guerraoui et al., PODC 2019).

Against that backdrop, Setu adds three things, and claims only these: (1) a very lightweight, dependency-free working implementation that is live and reproducible; (2) a server-enforced delegation-budget protocol (§5) that lets a principal cap an agent's cumulative spend — the mechanism agent commerce specifically needs and that a payment ledger alone does not provide; and (3) a commercialisation thesis (§7, §8) that anchors the engine to regulated money and existing agent standards. The strongest one-line framing: agent commerce may finally give consensusless-payment research a commercially valuable use case it previously lacked.

04Protocol and threat model

Setu instantiates the FastPay 3f + 1 authority model, tolerating up to f Byzantine authorities. The prototype uses f = 1 (four authorities, quorum three).

State

Each authority holds, per account, a balance and a monotonic nextSeq, plus at most one pending order. Accounts are addressed by an Ed25519 public key.

Transfer order and certificate

TransferOrder { sender, recipient, amount, seq, ref?, delegation? }
SignedOrder   { order, senderSignature }              // signed by the sender
Certificate   { order, senderSignature,
                authoritySignatures[] }               // a quorum of authority signatures

Payment flow

  1. Order. The sender signs a TransferOrder with seq = nextSeq and sends it to all authorities.
  2. Countersign. Each authority checks the signature, the sequence, and balance ≥ amount; if it holds no conflicting pending at this sequence, it locks the order as pending (first-seen) and returns its signature. Re-signing the identical order is idempotent.
  3. Certificate = a quorum of signatures. The sender assembles 2f + 1 signatures into a certificate.
  4. Confirm. The sender delivers the certificate to the authorities, which verify the quorum, debit the sender, credit the recipient, advance nextSeq, and clear the lock. Delivery to a quorum is what makes the received funds spendable (§6).

Safety (no double-spend)

Two conflicting orders at the same (sender, seq) would each need a quorum of 2f + 1 from 3f + 1 authorities. Any two such quorums intersect in at least f + 1 authorities, so their overlap contains at least one honest authority — which, by first-seen locking, signs only one of the two. A second conflicting certificate therefore cannot form. With f = 1, two three-of-four quorums overlap in two authorities, at least one honest.

Liveness, replay, recovery

The sender retries against reachable authorities; safety needs no timing assumption. Replay is prevented by per-account sequencing (a settled sequence is never re-applied). An authority that restarts or lags re-applies any certificate it is shown (idempotent settlement) and so catches up without a global synchronisation point. Per-account persistence includes the pending lock, without which a restarted authority could sign a conflicting order.

Fault and network assumptions

Safety holds under asynchrony with up to f Byzantine authorities. What the model does not give, and what therefore remain design work: authority replacement, key rotation, and checkpointing have no globally agreed switch-over point without a coordination mechanism — FastPay flags these as hard without consensus, and Setu accepts that a governance-layer consensus may be needed for those operations even though ordinary payments need none (§10). And in the prototype the four authorities share one operator, codebase, and cloud: that is geographic distribution with little common-mode independence, so the deployment today demonstrates replication, not decentralised accountability.

05Delegated budgets, enforced by the authorities

Agent commerce needs a principal to bound what an agent may spend. The subtle part — and a place a naive design breaks — is that a signed budget credential proves a permission was granted; it does not track how much has already been spent. If enforcement were left to the credential and offline merchants, an agent could present the same allowance to ten merchants at once, each verifying it as valid, and overspend. The budget must be live state held by the settlement authorities, not a static signature.

Setu therefore makes the authorities enforce a delegation-allowance object:

DelegationAllowance {
  id, principal, agent,        // who granted, who may spend
  total, spent,                // cumulative ceiling and running total (authority-held)
  maxPerPayment,
  validFrom, expiresAt,
  seq,                         // per-delegation sequence
  revoked                      // authority-held revocation flag
}

The principal signs the immutable grant (id, principal, agent, total, maxPerPayment, expiry) and registers it with the authorities; the authorities own the mutable spent, seq, and revoked fields. A delegated payment carries the delegation id; on each such payment an authority checks agent identity, expiry, per-payment cap, and spent + amount ≤ total, then debits the principal and advances spent. Because delegated spends are serialised by the delegation's own sequence and gated on the live total:

  • cumulative spend across any number of merchants cannot exceed total;
  • the "present the same budget to many merchants at once" attack yields at most one certificate, never two (concurrent conflicting orders at worst wedge the agent's own delegation sequence — the principal's other funds are untouched);
  • revocation is enforced by the authorities, so a revoked credential cannot be spent even against a merchant that never saw the revocation.

This is implemented and demonstrated (npm run demo:allowance). It is the mechanism that makes "your key holds the leash" a settlement-level guarantee rather than a client-side hope.

06Prototype and evidence

~4 mscertificate finality, one machine
~200 mscertificate finality, four regions
280/280chained spends first-try, no extra delay

Four authorities run on Fly.io in London, Frankfurt, Washington and Singapore (four regions, three continents). A browser wallet at setu-mocha.vercel.app lets anyone create a keypair, draw test units, and settle a payment across those regions directly from the browser; a public explorer streams live transactions.

Adversarial safety. A double-spend cannot form a certificate; a Byzantine authority that signs everything cannot enable one; a killed authority process does not halt settlement; a delegated budget cannot be over-spent, double-presented, or used after revocation.

Chained-spend (the benchmark that matters). "Final in one round trip" is certificate-receipt finality; the commercially decisive question is whether a recipient can immediately spend received funds. A dedicated benchmark runs 40 concurrent eight-hop chains, each hop after the first spending the money the previous hop delivered: all 280 chained spends completed first-try, at no extra latency over a fresh payment, and the property held with an authority killed (money delivered to the surviving quorum stayed immediately spendable). The same benchmark run against the live four-region network gives 48 of 48 chained spends first-try at ~183 ms per hop (p50) — statistically the same as a fresh payment (delta +24 ms). That is spendable-balance finality, measured both on a laptop and across the wide area; the authority-failure case is validated in the localhost run.

What the numbers do not establish. 423 transactions/second on a laptop shows the software runs, not production scale; FastPay's 80,000 TPS came from favourable lab conditions, lightly-loaded WAN authorities, no full persistent-database costs, and independent accounts (repeated spends from one sender cannot be parallelised). Persistence, contention, recovery, and security controls are part of the production design, so throughput is a design capability the prototype has not demonstrated.

07Production architecture: anchoring to real money

The single most important open question is what a unit represents. The live network settles test units. Moving a number is easy; the legal and financial claim behind it is the hard part, and it is not a coding problem.

The BIS distinguishes an operational ledger transfer from legal settlement finality, which is a status conferred by law and contract at a defined point — a protocol cannot assert it about itself. International standards prefer settlement in central-bank money, or a settlement asset with tightly controlled credit and liquidity risk. Setu is deliberately asset-agnostic: it supplies fast agreement among accountable parties and a verifiable receipt, and must be anchored to a defined claim. One concrete reference architecture, end to end:

  1. A regulated issuer (e-money institution, licensed stablecoin issuer, or bank) receives fiat and holds it in a segregated, safeguarded omnibus account or reserve.
  2. A Setu balance represents a defined beneficial claim against that pool. Payments move those claims on the Setu ledger — fast, feeless, internal.
  3. Mint and redeem happen only through the regulated issuer: fiat/stablecoin in creates Setu units; redemption burns them.
  4. Periodic reconciliation settles the omnibus position against the banking or on-chain layer; intraday Setu transfers net down to far fewer external settlements.
  5. The legal rules state when a payer's obligation is discharged, and the safeguarding and insolvency terms state holders' redemption rights if the issuer fails.

This makes Setu an off-chain, low-latency transfer layer over claims on regulated money — deliberately, and stated plainly so a reader can evaluate it. It also bounds the prefunding cost: every payment is prefunded, which removes credit and liquidity management but forgoes credit, netting, overdrafts, delayed capture and FX, and gross-settles rather than nets — attractive for micropayments, less so where working-capital efficiency dominates.

08Market position

Agent commerce is no longer an empty field. The layers are separating, and Setu occupies one of them:

LayerRoleExamples
Intent & authorityWhat the user authorised; accountabilityAP2 (now with the FIDO Alliance)
Payment requestHTTP request/response for a paymentx402
Settlement backendMove the claim, fast, with enforced limitsSetu
Monetary claimLegal money and redemptionRegulated stablecoin / e-money issuer
Integrated incumbentSeveral layers in one networkMastercard Agent Pay for Machines

The principal commercial challenge is Mastercard Agent Pay for Machines (announced June 2026), which targets exactly this category — high-frequency, low-value machine payments with agent credentialing, programmatic spend limits, and multi-rail settlement across cards, accounts and stablecoins. Lower latency is not a sufficient answer. A cloud platform, API provider, or bank would choose Setu over an incumbent network only for concrete reasons: materially better economics on sub-cent transactions; self-hosting or consortium control rather than dependence on a proprietary network; deterministic settlement without card-authorisation semantics; no chargeback exposure on completed digital services; stronger, auditable programmatic limits (§5); data minimisation; or deployment inside a controlled ecosystem the operator already owns. Where none of these dominate, the incumbent's money, acceptance, and network effects will — and Setu should not pretend otherwise. Payments are adopted on acceptance and interoperability, which is why Setu's roadmap is to interoperate (x402/AP2) rather than to replace. A first step is live: an x402-style gateway (setu-gateway.fly.dev) lets any agent pay a web resource over Setu with no Setu-specific code, discoverable via an A2A-style agent card, and an MCP server lets any MCP agent hold a Setu wallet directly.

A related bound Setu does not solve: a payment receipt is not a completed transaction. It proves an agent paid an invoice, not that the service was delivered, correct, or refundable. Those need escrow, dispute handling, or conditional execution — several of which are multi-party atomic operations where an intermediary or consensus re-enters (FastPay likewise does not support payment-versus-payment or delivery-versus-payment). For pay-then-consume API calls this is adequate; for goods, travel, or regulated products it is not, and Setu should be deployed only where "pay, receive, stop using a bad provider" is a sufficient dispute model, or behind a marketplace that supplies the rest.

11References, reproducibility, acknowledgements

Reproducibility

The prototype is dependency-free TypeScript on Node. The claims in §5–§6 are reproducible:

npm run demo            # settlement safety: double-spend, Byzantine authority, outage
npm run demo:allowance  # server-enforced delegated budgets (§5)
npm run bench:chain     # chained-spend benchmark (§6)
npm run bench           # throughput benchmark

Source: github.com/rainaajay/setu (MIT). Deployment configuration and benchmark scripts are in the repository. Independent cryptographic and distributed-systems review is still outstanding (§10).
